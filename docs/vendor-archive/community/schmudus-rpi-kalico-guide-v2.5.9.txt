Step 1: Install a Raspberry Pi image
use Raspberry Pi imager
Select Raspberry Pi OS other --> Raspberry Pi OS lite (64-bit or 32-bit)
Set network and SSH using “pi” and “raspberry”

IMPORTANT
BTT SKSM is NOT working with Raspberry Pi 5 (too much power needed for Pi 5)

It is necessary to use a high power (5 Amps) power device! The original Raspberry power supply is a little
weak if you want to run several USB devices. There is a 45W power supply available which is better.
Use a separate power supply for each HAT
The Pi 5 should boot directly from NVMe without SD card, so you can use Pi Imager to write image directly
to your NVMe and install it to the Raspberry hat.
After finishing the steps and your Pi is booting just fine we continue.

sudo apt autoremove

Change boot order if using NVMe

sudo -E rpi-eeprom-config --edit
change boot order to:
BOOT_ORDER=0xf146
If not already add:
PCIE_PROBE=1
Save and reboot.
INFO: You can change boot order via raspi-config as well, but it will maybe not add the PCIE_PROBE=1.
Some NVMe will not start properly without this order.

If you like to change the cooling fan speed you need to change the boot/firmware/config.txt
Add e.g. the following:
[all]
dtparam=fan_temp0=30000

dtparam=fan_temp0_speed=65
dtparam=fan_temp1=55000
dtparam=fan_temp1_hyst=5000
dtparam=fan_temp1_speed=128
dtparam=fan_temp2=75000
dtparam=fan_temp2_hyst=5000
dtparam=fan_temp2_speed=154
dtparam=fan_temp3=80000
dtparam=fan_temp3_hyst=4000
dtparam=fan_temp3_speed=192
dtparam=fan_temp4=90000
dtparam=fan_temp4_hyst=4000
dtparam=fan_temp4_speed=255
# add this for enabling maximum current for usb
usb_max_current_enable=1
#add this for PCIe in V3 mode

dtoverlay=disable-bt

Disable WIFI power saver

check status with:
sudo iwconfig wlan0
Disable power saver with
sudo crontab -e
which opens the root crontab in your chosen editor... (or select your editor, usually nano)
add the following line at the bottom of the root crontab:
@reboot /usr/sbin/iw wlan0 set power_save off
Save, exit and reboot. Check again with sudo iwconfig wlan0 if it is disabled.

Change file permissions
Now, we need to change the file permissions to make it readable by all by typing
sudo chmod 644 /etc/systemd/system/magneto.service
As the last step, you need to tell the system that you have added this file and want to enable this service so that it
starts at boot.
sudo systemctl daemon-reload
sudo systemctl enable magneto.service
change can config (if using CAN):
sudo nano /etc/network/interfaces.d/can0
----paste this:
allow-hotplug can0

txqueuelen 1024 can work, but Kevin recommends to use txqueuelen 128, I set it to 512.
see also https://www.klipper3d.org/CANBUS.html

Extruder stepper motor (only nitehawk36)

The nitehawk36 board delivers a sense_resistor of 0.100, so change it in your config
[tmc2209 extruder]
uart_pin: nhk:gpio0
interpolate: true
run_current: 1.0
#run and hold current are depending on motor you have.
#Keep it on the original values if you don't change the motor
#Additional cooling (cooling fan e.g.) is necessary with current higher than 0.7
hold_current: 0.6
sense_resistor: 0.100
stealthchop_threshold: 0
Additional cooling fan

The stands on the magneto toolhead can fit a 40 x 40 cooling fan. It can be wired to nitehawk36 Probe port,
using GND and PRB+. Its always on, so a quite fan is recommended (Noctua e.g.). Voltage can be set to 24 or
5V by jumper.

Add CAN information to network:
Go to etc\network
create a directory called interfaces.d (sudo mkdir interfaces.d)
open the new directory
make two files:
sudo nano can0
insert:
auto can0
allow-hotplug can0
iface can0 can static
bitrate 250000
up ip link set $IFACE txqueuelen 512
close and safe

next file is very similar:
sudo nano can1
insert:
auto can1
allow-hotplug can1
iface can1 can static
bitrate 250000
up ip link set $IFACE txqueuelen 512
reboot with sudo reboot
sudo systemctl enable systemd-networkd
sudo systemctl start systemd-networkd
sudo systemctl disable systemd-networkd-wait-online.service
next is one row:
echo -e 'SUBSYSTEM=="net", ACTION=="change|add", KERNEL=="can*" ATTR{tx_queue_len}="512"' | sudo tee
/etc/udev/rules.d/10-can.rules > /dev/null
next is one row:
echo -e "[Match]\nName=can*\n\n[CAN]\nBitRate=250000\nRestartSec=0.1s\n\n[Link]\nRequiredForOnline=no" |
sudo tee /etc/systemd/network/25-can.network > /dev/null
After another sudo reboot and with ip a you should see something like this: (ip a is a shortcut for ip address)
[...]
4: can0: <NOARP,UP,LOWER_UP,ECHO> mtu 16 qdisc pfifo_fast state UP group default qlen 512
link/can
5: can1: <NOARP,UP,LOWER_UP,ECHO> mtu 16 qdisc pfifo_fast state UP group default qlen 512
link/can
State UP means, its running :-)
CAN txqueuelen different value after reboot
If you see the txqueuelen as a different number than set try the following (group default qlen set it to different value):
check:
etc/udev/rules.d/10-can.rules
etc/udev/rules.d/10-can-queue.rules
Both should look like this:
SUBSYSTEM=="net", ACTION=="change|add", KERNEL=="can*" ATTR{tx_queue_len}="512"

How to find CAN UUID
Stop Klipper and query the corrssponding CANBUS (CAN0 / CAN1)
sudo systemctl stop klipper.service
~/klippy-env/bin/python ~/klipper/scripts/canbus_query.py can0
~/klippy-env/bin/python ~/klipper/scripts/canbus_query.py can1
sudo systemctl start klipper.service
It may be possible that you have to do a complete powercycle before changes are working. EBB36 seems to need a
powercycle before new firmware takes part.
